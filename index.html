Overview

What is Agentic AI?

You've probably used generative AI like ChatGPT, Grok, or Claude on the web—you ask a question, it gives an answer. That's powerful, but it's just conversation. You still have to take the answer and do something with it yourself.

"Agentic AI" is different. It can take actions on your behalf—reading files, writing code, running programs, and completing multi-step tasks autonomously. Instead of just answering questions, Claude CLI can actually do things: analyze your data, build tools, automate workflows, and more.

You describe what you want in plain English. Claude figures out the steps, writes any necessary code, runs it, and delivers results. You guide the process with feedback, but Claude handles the technical details.

No coding experience required. You'll learn to describe tasks clearly and review results—Claude handles the programming.
You'll need admin rights on your computer to install the required software. If you're on a company laptop with restricted permissions, check with IT first.

What You Can Do

Data Analysis

Read Excel/CSV files, identify trends, segment data, and generate actionable insights

"Read this customer sales history by product. Study the data thoroughly, then act as a product and data analyst to identify sales trends by customer, segment, and product. Develop a prioritized action plan to address declines. Generate results in separate tabs with both qualitative insights and quantitative analysis."

Automation

Batch process files, transform data between formats, schedule recurring tasks

"I have 50 invoice PDFs in this folder. Extract the vendor name, invoice number, date, and total from each one. Create an Excel summary with all invoices sorted by date, and flag any duplicates."

Document Processing

Parse contracts, extract key terms, compare versions, generate summaries

"I have three vendor contracts in this folder. Create a comparison matrix showing: pricing terms, SLA commitments, termination clauses, and liability limits. Highlight any unusual or concerning terms and recommend which vendor offers the best overall value."

Tool Building

Create custom scripts, connect to APIs, build dashboards and reports

"Build me a simple web dashboard that connects to our Airtable base, pulls the sales data, and shows a chart of monthly revenue with the ability to filter by region. Make it look professional."

Research & Review

Analyze documents, research alternatives, provide recommendations with evidence

"Review this vendor agreement and service charges. Research the web and discussion boards to find how others have negotiated these terms more favorably. Provide real examples and commentary we can use to redline the agreement. Be thorough versus fast."

Problem Solving

Debug issues, troubleshoot processes, find root causes and solutions

"My monthly report script stopped working after IT updated our systems. Here's the error I'm getting. Figure out what changed, fix it, and make sure it won't break again. Test it thoroughly before telling me it's done."

The Four Components You'll Set Up

VS Code

Your workspace. A free editor that shows your files on the left, content in the middle, and has a built-in Terminal where you'll interact with Claude.

Python

The engine. A programming language that Claude uses to build solutions. You don't need to learn it—Claude writes it—but it needs to be installed.

Claude CLI

Your AI partner. The command-line interface for Claude. You describe what you want, Claude reads your files, writes code, runs it, and delivers results.

GitHub

Your safety net. Cloud storage that tracks every change to your work. If something breaks, you can always go back.

Phase 1: Foundation Setup

30-45 minutes Not started

Install the basic tools and get comfortable with VS Code's layout. You can stop after this phase and return later.

Already have some tools installed? Run the verification command for each step. If it works, skip to the next step.

Step 1.1: Install VS Code

  1. Open your browser and go to: code.visualstudio.com/Download
  2. Click the big blue Mac button to download
  3. Wait for the download, then double-click the .zip file to extract it
  4. Drag "Visual Studio Code" to your Applications folder
  5. Run the downloaded installer (VSCodeUserSetup-x64-x.xx.x.exe)
  6. Accept the license agreement and click Next
  7. IMPORTANT: On "Select Additional Tasks," check ALL boxes, especially "Add to PATH"
  8. Open VS Code: Go to Applications folder and double-click Visual Studio Code
Success looks like: A window opens with "Welcome" tab. You'll see an interface with icons on the left side.

Step 1.2: Learn the VS Code Layout

Spend 5 minutes understanding the interface—it'll make everything easier.

Activity Bar Far left icons. The files icon (Explorer) is most important.
Sidebar Shows your folders and files, like Finder/File Explorer.
Editor Area Where you view and edit files. Files open in tabs.
⭐ Terminal THE IMPORTANT ONE. You'll interact with Claude here. Press Ctrl+` to toggle.

Try it now:

  1. Press Ctrl+` (backtick key, left of 1). Terminal appears at bottom.
  2. Press Ctrl+` again. Terminal hides.
  3. Press Ctrl+` once more. Click in the Terminal area.

Step 1.3: Install Homebrew

Homebrew is like an "app store" for developer tools on Mac. Instead of downloading installers from different websites, you can install most tools with a single command. It handles all the complexity for you.

  1. Open VS Code with Terminal visible (Ctrl+`)
  2. Copy and paste this command, then press Enter:
Terminal 
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
  1. Enter your Mac password when prompted (you won't see characters—that's normal)
  2. Press Enter when asked to continue
  3. Wait 2-5 minutes for installation
  4. CRITICAL: Look for "Next steps" at the end and run those commands!
Don't skip the "Next steps" commands! Copy and run what YOUR terminal shows—the exact path depends on your Mac (Apple Silicon uses /opt/homebrew, Intel uses /usr/local).
  1. Restart your terminal: Close the terminal tab (click X) and open a new one (Ctrl+`)

Verify Homebrew works:

Terminal 
brew --version
Success: You see "Homebrew 4.x.x"
Troubleshooting

"command not found" - You missed the "Next steps." Close Terminal, reopen, and run the commands shown at the end of Homebrew installation.

Step 1.4: Install Python

Use Homebrew to install Python:

Terminal 
brew install python
  1. Go to: python.org/downloads
  2. Click the yellow "Download Python 3.x.x" button
  3. Run the downloaded installer
  4. CRITICAL: Check "Add Python to PATH" at the bottom!
  5. Click "Install Now"
If you miss "Add Python to PATH," uninstall and reinstall with that box checked.

Verify installation (wait for install to finish):

Terminal 
python3 --version
Success: You see "Python 3.12.x" or "Python 3.13.x"
On Mac, always use python3 (not just python).

Step 1.5: Install Git

Git tracks changes to your work—like version history for your files.

Terminal 
brew install git
  1. Go to: git-scm.com/download/win
  2. Download should start automatically
  3. Run installer, click Next through screens (defaults are fine)
  4. On "Choosing the default editor," select VS Code if available
  5. Click Install, then Finish
  6. Close and reopen VS Code

Verify Git installed:

Terminal 
git --version

Configure Git with your name and email:

Terminal 
git config --global user.name "Your Name"

Replace "Your Name" with your actual name

Terminal 
git config --global user.email "your.email@example.com"

Use the same email you'll use for GitHub

Step 1.6: Create Projects Folder

Create a dedicated folder for all your projects:

Terminal 
mkdir -p ~/Documents/projects

mkdir = "make directory" (creates a new folder)

Open this folder in VS Code:

  1. Click File → Open Folder (or Cmd+O on Mac / Ctrl+K Ctrl+O on Windows)
  2. Navigate to Documents → projects
  3. Click Open
Success: You see "PROJECTS" in the VS Code sidebar (it will be empty).

Inside Claude vs. Terminal
You're now inside Claude—your prompt shows > waiting for your request. This is different from the regular terminal (which shows $ or your username).

Step 2.5: Your First Claude Session

Let's practice working both inside and outside Claude so you're comfortable with both.

Part A: Working in the Terminal (outside Claude)

First, exit Claude if you're still inside it:

Inside Claude 
/exit

Now you're back in the regular terminal. Create a test project folder using terminal commands:

Terminal (outside Claude) 
cd ~/Documents/projects && mkdir my-first-project && cd my-first-project

cd = change directory, mkdir = make directory. The && chains commands together (runs each if the previous succeeds).

Part B: Working with Claude (inside Claude)

Now start Claude to work on your project:

Terminal (outside Claude)

Once you see Claude's > prompt, try your first request:

Inside Claude

Watch what happens:

  3. You see the file appear in VS Code's sidebar

Inside Claude

Phase 3: Project Structure

20-30 minutes Not started

Learn how to organize projects so Claude works better and your work stays organized.

When to create a new project vs. reuse an existing one:
  • New project + venv: Building something distinct (a dashboard, an app, a tool)
  • Reuse existing project: One-off scripts, quick analyses, related tasks that share data or logic

Example: If you built a "sales-analysis" project last week and now want to do a similar analysis, just open that folder and work there. No need to create a new project for every task.

Step 3.1: Create Virtual Environment

A "virtual environment" (venv) is an isolated workspace for each project. Think of it like having separate toolboxes for different jobs—your kitchen renovation tools don't mix with your gardening tools. Setting one up is easy and takes seconds.

Terminal (outside Claude) 
python3 -m venv venv

Activate it:

Terminal (outside Claude) 
source venv/bin/activate
Success: You see "(venv)" at the start of your Terminal prompt.
Remember: Activate your venv every time you start working on a project!
PowerShell execution policy error?

Run this once:

PowerShell 
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

Then try activating again.

Step 3.2: Create .claude Configuration

The .claude folder gives Claude context about your project. Claude reads CLAUDE.md automatically when you start a session.

Terminal (outside Claude) 
mkdir -p .claude

Download these template files:

Files for the .claude folder

File Purpose
CLAUDE.md Project overview Claude reads automatically at session start. Download
AI_CODING_GUIDELINES.md Rules for how Claude should work. Download
SESSION_LOG.md Prevents amnesia. Update at the end of each session. Download

Files for the project root

File Purpose
README.md Quick reference for yourself. Download
.gitignore Tells Git which files to ignore. Download
If .gitignore downloads with the wrong name

The file must be named exactly .gitignore (with the leading dot). To rename it:

Mac: In Terminal, run: mv ~/Downloads/gitignore ~/Downloads/.gitignore

Windows: In PowerShell, run: ren $env:USERPROFILE\Downloads\gitignore .gitignore

Then move the renamed file to your project folder.

Move the files into place:

  1. Open your Downloads folder in Finder
  2. Drag CLAUDE.md, AI_CODING_GUIDELINES.md, and SESSION_LOG.md into the .claude folder in VS Code's sidebar
  3. Drag README.md and .gitignore into the project root (the top level, not inside .claude)
Success: You see the new files in your project sidebar (.claude folder with 3 files, plus README.md and .gitignore).

Phase 4: GitHub Integration

20-30 minutes Not started

Connect to GitHub for backup and version history. If something breaks, you can always go back.

This phase is not optional. GitHub is your undo button. Claude will make changes to your files—sometimes many at once. Without version control, a mistake could mean losing hours of work. With GitHub, you can always roll back.

What is GitHub?

GitHub is like Google Drive for code—but smarter. It stores your project files online and tracks every change you make. If you accidentally break something, you can rewind to any previous version. It's free for personal use and used by millions of developers worldwide.

Repository (Repo)

A project folder stored on GitHub. Each project gets its own repo.

Local vs Remote

Local = files on your computer. Remote = the copy on GitHub. You work locally, then "push" changes to the remote.

Commit

A snapshot of your project at a point in time, with a note about what changed. Like "Save As" with a description.

.gitignore

A file that tells Git which files to NOT track (passwords, large data files, virtual environments).

Why bother? Claude makes lots of changes fast. GitHub lets you track what changed, when, and why. If Claude breaks something, you can see exactly what happened and undo it.

Step 4.1: Create GitHub Account

  1. Go to: github.com
  2. Click "Sign up"
  3. Choose a username (visible publicly)
  4. Use the same email you configured Git with
  5. Verify your email

Step 4.2: Install GitHub CLI

Terminal (outside Claude) 
brew install gh
  1. Go to: cli.github.com
  2. Click "Download for Windows"
  3. Run installer, accept defaults
  4. Close and reopen VS Code

Authenticate:

Terminal (outside Claude) 
gh auth login

Follow prompts: GitHub.com → HTTPS → Login with web browser → Copy code → Authorize

Success: You see "Logged in as your-username"

Step 4.3: Save to GitHub

Your .gitignore file from Phase 3 tells Git which files to ignore. Now let's initialize Git and save your project.

If you're inside Claude, exit first:

Inside Claude 
/exit

Set up Git and make your first commit:

Terminal (outside Claude) 
git init && git add . && git commit -m "Initial project setup"

git init = start tracking this folder, git add . = stage all files, git commit = save a snapshot with a message

Create repository on GitHub and push:

Terminal (outside Claude) 
gh repo create my-first-project --private --source=. --push

gh repo create = create a new repository on GitHub from this folder and upload it

--private means only you can see it. Change to --public to share.

From Theory to Reality: What Actually Works

Anyone can get Claude to produce something. The difference between "okay" results and genuinely useful output comes down to how you set up and guide the work. Here's what separates success from frustration:

Start Strong: Spell Out Everything Up Front

Before Claude writes a single line, clearly describe:

  • The need — What problem are you solving? Why does this matter?
  • The functionality — What should it do, step by step?
  • The end result — What does "done" look like? What output do you expect?
  • The usage — Who will use this? How? How often?

This upfront investment pays off massively. Claude makes better decisions when it understands the full picture.

For larger projects, consider writing a PRD (Product Requirements Document) — a structured brief that captures all requirements in one place. Claude can help you write one. See PRD template →

Review Like a Manager

After Claude completes work, don't just accept it. Ask Claude to review its own work as if it were a manager:

"Now review this as if you were a senior app development manager. Evaluate the design, the code quality, and the results. What would you improve?"

You can tailor the reviewer role: CFO for financial analysis, data architect for database design, UX expert for user-facing tools. This catches issues Claude missed the first time.

Partner, Not Yes-Man

Tell Claude explicitly: "Be my partner, not a yes-man." You want Claude to push back when something doesn't make sense, suggest better approaches, and challenge assumptions. A good partner tells you when you're heading in the wrong direction—before you've wasted hours.

Never Assume

Tell Claude: "If you're unsure about anything, ask me rather than guessing." Assumptions lead to rework. Questions lead to clarity.

DRY — Don't Repeat Yourself

If Claude is duplicating logic or creating similar code in multiple places, call it out. One piece of code, one source of truth.

SSOT — Single Source of Truth

Data should live in one place. If Claude creates multiple copies of information, it will eventually get out of sync. Insist on one authoritative source.

Iterate and Refine

First results are rarely perfect. Say "that's close, but can you also..." to guide Claude toward exactly what you need.

Tips & Best Practices

Additional guidance for getting the most out of Claude.

First, the fundamentals. The tips below will help, but they won't matter much if you skip the core principles. Read "From Theory to Reality" →

Additional Tips

Be Specific About Quality

Add phrases like "be thorough versus fast" or "test it thoroughly before telling me it's done." Claude will put in more effort when you signal that quality matters.

Ask for the Full Solution

Instead of accepting "you'll need to..." responses, say: "Don't just tell me what to do—do it for me." Claude can often complete tasks it initially just explains.

Provide Context

Share relevant files and explain the bigger picture. "This is for a quarterly board report" helps Claude make better decisions than just "analyze this data."

APIs and Environment Variables

As you build more powerful tools, you may want to connect to external services (Airtable, HubSpot, Google Sheets, etc.). These connections use API keys—secret passwords that grant access.

Never put API keys directly in your code. Anyone who sees your code would have your keys. Instead, store them in a .env file.

Create a file called .env in your project root:

.env file 
# My API Keys (never commit this file!)
AIRTABLE_API_KEY=your_key_here
OPENAI_API_KEY=your_key_here
HUBSPOT_API_KEY=your_key_here

Your .gitignore file (from the templates) already includes .env, so it won't be uploaded to GitHub. Ask Claude to help you set up any API connections—it will use the .env pattern automatically.

Refactoring Existing Code

When you need to restructure or improve an existing project, Claude can help—but refactoring requires a disciplined approach:

Understand First

Before changing anything, ask Claude to explain the current architecture. "Walk me through how this codebase is structured and how the pieces connect."

Plan Before Coding

Document what you're changing and why. Ask Claude to create a refactoring plan with clear steps before touching any code.

One Change at a Time

Refactor incrementally. Make one change, test it works, commit it. Then move to the next. Never leave the codebase broken.

Don't Mix with Features

Refactoring and adding features are separate tasks. Do one or the other, not both at once. Mixing them makes bugs hard to trace.

Commit frequently during refactoring. Small commits make rollback easy. If something breaks, you can undo just the last change instead of losing hours of work.

Safety Basics

Claude is powerful—which means a few simple habits keep your work secure:

Protect API Keys

Keep them in .env files, never in code. Your .gitignore prevents them from being uploaded.

Review Before Approving

Claude asks permission before running commands or making changes. Take a moment to understand what it's about to do.

Commit Often

Regular Git commits mean you can always undo changes. If something goes wrong, you have a safe point to return to.

Keep Projects Separate

One folder per project, one venv per project. This prevents Claude from accidentally affecting other work.
